From Vibe Coding to Agentic Engineering

Since GPT launched, asking LLMs coding questions has been common. Early on they were unreliable, and everything involved copy/pasting between a chat window and your editor. That changed when tools like Cursor and VS Code integrated AI directly into the development environment, giving them more power to handle real tasks. This brought on a label that is objectively inaccurate, but stuck: Vibe Coding.

It coincided with the launch of Claude Code, which started as a small research project at Anthropic built in the command line, but has now completely reshaped software development. As its creator explains, it "now writes 100% of its own code." These have gone from toys to tools that software engineers actually use daily.

That does not mean you have to use them, but you do need to know they exist and understand their basics. It also matters how you use them, as shown by Anthropic's own research on AI-assisted coding.

The biggest shift is that instead of writing prompts, the focus is now on creating clearly structured files that describe your project, its components, and testing methods. This practice is built around industry standards used by development teams. The core skill is the same whether you use AI tools or not: clearly documenting what you want to build. That is what this workshop is about.

Your PRD should be a Markdown file (.md)

Markdown is an open plain-text format that allows for hierarchically structured text. You probably know it from GitHub README files. PRDs written in Markdown are readable by both humans and AI coding agents. Create a plain text file with the .md extension (e.g. PRD.md) and use headings, lists, and sections to organize your requirements.

If you plan to use AI coding tools: set up VS Code and GitHub Copilot using this guide: VS Code Setup Guide. You get Copilot Pro free with your OCAD email. Other tools (Claude Code, Cursor) work similarly but may require separate accounts.

Finding Your Focus

You are already mid-project. The goal is to identify a specific piece of your thesis that would benefit from clear technical documentation. Here are some strategies for finding that focus:

Format: Individual or groups of 2-3

The Thing You Keep Avoiding

There is probably a technical piece of your project that you know needs to happen but have been putting off. That avoidance usually means the scope is unclear. Documenting it as a PRD forces you to define the problem clearly.

The Next Feature

What is the next thing you need to add or build? Write the PRD for that specific feature or system, not the entire project.

A Tool That Helps You Work

A utility, generator, converter, or helper that would speed up your thesis development. Think: asset generators, data processors, testing tools, calibration utilities, or batch processors.

Something Beyond What You Can Currently Build

A clear PRD lets you attempt things that are currently outside your skill set. If you can describe it precisely enough, you can build it with an AI coding agent -- or use the documentation to learn how to build it yourself.

Workshop Steps

Step 1: Identify Your Focus (5 min)

Pick one specific piece of your thesis project to document. See "Finding Your Focus" above for strategies. Scope it down to something concrete -- a single feature, system, or tool rather than the whole project.

Step 2: Brain Dump (10 min)

Write down everything you know about what you want to build in plain language. Do not worry about structure -- capture features, constraints, uncertainties, and preferences. Get it all out.

Step 3: Structure into a PRD (15 min)

Use the Project Brief template from the AI Coding Resources page to organize your brain dump into a Markdown (.md) file. You can also ask an AI agent to help structure it, but review what it produces -- you know your project better than it does.

Step 4: Peer Review (10 min)

Exchange documents with a partner. Can they understand what you are building from the document alone? What questions do they have? What is unclear or missing?

Step 5: Refine Your PRD (15 min)

Based on feedback, fill gaps and clarify ambiguities. Add explicit constraints, examples, and edge cases. The PRD should be detailed enough that someone unfamiliar with your project could understand what needs to be built.

Step 6 (Optional): Test with a Coding Agent (remaining time)

If you want to try building with your PRD, use it to guide development with Copilot, Claude Code, or another coding agent. This step is not required -- the PRD itself is the primary deliverable. But if you do try it, notice where your documentation helps and where you need to clarify further.

Deliverable

Submit a post containing:

  1. Your PRD (.md file) — The structured project documentation you created, as a Markdown file (can be linked or pasted as text)
  2. Brief reflection — 2-3 sentences on how the documentation process affected how you think about your project
  3. (Optional) Link to your project/tool — If you used your PRD with a coding agent, include a link to the working prototype, GitHub repo, or hosted version

Project Ideas by Medium

Not sure what to build? Here are starting points based on your thesis type:

Games / Interactive Experiences

  • A core mechanic prototype (one interaction loop, polished)
  • Level generator or asset randomizer
  • Save/load system or state manager
  • Dialogue or quest system tool

Installations / Physical Computing

  • Sensor data visualizer or debugger
  • Calibration tool for projections or sensors
  • State machine for installation modes
  • Data logger for user interaction patterns

Visual / Motion Media

  • Batch image processor or resizer
  • Color palette generator from reference images
  • SVG or graphics generator
  • Animation timing calculator

UX / App Projects

  • Interactive prototype (beyond Figma)
  • Data visualization dashboard
  • Form or survey with custom logic
  • API integration proof of concept
